# Security policy

qslib is pre-release research software. No version is currently supported as a
security-stable release.

## Reporting a vulnerability

Do not disclose a suspected vulnerability in a public issue. Once the dedicated
repository is established, use its private vulnerability-reporting facility.
If that facility is unavailable, contact the repository owner through a private
channel shown on the repository profile and ask for a reporting address without
including exploit details.

Include the affected revision, platform, minimal reproduction, expected impact,
and whether the issue has been disclosed elsewhere. The maintainer will
acknowledge a complete report, assess affected versions, and coordinate a fix
and disclosure timeline. No fixed response deadline is promised before 1.0.

## Scientific integrity and software security

Incorrect physical results, silent convention changes, malformed artifact
acceptance, unsafe foreign-memory access, and reproducibility failures may be
security or integrity issues when they can corrupt trusted workflows. Report
them privately when public reproduction could expose sensitive data or create
an exploitable condition. Ordinary numerical defects can use the public issue
tracker after the repository exists.

Security reports never authorize access to systems, data, or credentials that
the reporter does not own or have permission to test.
